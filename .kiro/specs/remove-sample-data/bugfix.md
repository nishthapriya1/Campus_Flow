# Bugfix Requirements Document

## Introduction

The application's server startup logic automatically seeds sample/demo notifications and notices into MongoDB via `seedDatabase()` in `src/scripts/seed.js`. This seed function is invoked every time the server starts (when the admin user doesn't exist yet, or after a database reset). The seed script deletes all existing notifications and notices for the seeded accounts before reinserting hardcoded demo data, causing:

1. Sample notifications and notices always appear as if freshly created
2. After any server restart, demo data reappears even if previously deleted by the user
3. The application looks like a demo rather than a production system

Additionally, the seed script creates notifications with no relationship to actual system events, polluting the notification feed with fake data that mixes with legitimate notifications generated by the reminder scheduler and Guardian AI.

## Bug Analysis

### Current Behavior (Defect)

1.1 WHEN the server starts and the admin user does not exist THEN the system runs `seedDatabase()` which inserts 5 hardcoded sample notifications into MongoDB (deadline reminders, attendance risks, placement opportunities that are fabricated)

1.2 WHEN the server starts and the admin user does not exist THEN the system runs `seedDatabase()` which inserts 4 hardcoded sample notices into MongoDB (semester registration, capstone guidelines, sports fest, Google placement drive with fake S3 keys)

1.3 WHEN the server starts and the admin user already exists THEN the system skips seeding, but previously seeded sample notifications and notices persist in the database from the initial seed run

1.4 WHEN a user views the Notifications page THEN the system displays sample/demo notifications (e.g., "Math Exam Deadline Approaching", "Chemistry Lab Exam 24h Reminder", "Attendance Risk: Physics Class") that were not generated by actual system events

1.5 WHEN a user views the Notices page THEN the system displays sample/demo notices (e.g., "Semester_Registration_Notice.txt", "Capstone_Project_Guidelines.pdf") with fake S3 keys that do not correspond to real uploaded files

1.6 WHEN the database is reset or the admin user is removed THEN the system re-runs `seedDatabase()` on next startup, deleting all legitimate notifications and notices and replacing them with hardcoded demo data

### Expected Behavior (Correct)

2.1 WHEN the server starts THEN the system SHALL NOT insert any sample notifications into MongoDB; the seed script shall only create user accounts (admin and student) without inserting demo notifications

2.2 WHEN the server starts THEN the system SHALL NOT insert any sample notices into MongoDB; the seed script shall only create user accounts without inserting demo notices

2.3 WHEN the server starts and the admin user already exists THEN the system SHALL skip seeding entirely and SHALL NOT delete or modify any existing notifications or notices

2.4 WHEN a user views the Notifications page and no notifications exist THEN the system SHALL display an empty state message ("No notifications have been received.") without generating or injecting sample data

2.5 WHEN a user views the Notices page and no notices exist THEN the system SHALL display an empty state message ("No notices have been uploaded yet.") without generating or injecting sample data

2.6 WHEN the database is reset or the admin user is removed THEN the system SHALL only re-create user accounts on next startup without inserting any demo notifications or notices

### Unchanged Behavior (Regression Prevention)

3.1 WHEN the server starts and the admin user does not exist THEN the system SHALL CONTINUE TO create the default admin and student user accounts via `seedDatabase()`

3.2 WHEN the reminder scheduler detects an event entering the 24h or 1h window THEN the system SHALL CONTINUE TO generate legitimate notification documents via `fireReminderNotification()`

3.3 WHEN the 72h deadline scheduler detects upcoming exams/assignments THEN the system SHALL CONTINUE TO generate legitimate deadline notification documents via `runDeadlineCheck()`

3.4 WHEN the Guardian AI analysis produces alerts THEN the system SHALL CONTINUE TO create legitimate notification documents in the `/api/guardian/analyze` route

3.5 WHEN an administrator uploads a notice file via the admin portal THEN the system SHALL CONTINUE TO create a notice document in MongoDB via the `POST /api/notices` route

3.6 WHEN the Notifications page fetches from `GET /api/notifications` THEN the system SHALL CONTINUE TO return paginated results sorted by `createdAt` descending with unread count

3.7 WHEN the Notices page fetches from `GET /api/notices` THEN the system SHALL CONTINUE TO return paginated results sorted by urgency and upload date

3.8 WHEN the server starts THEN the system SHALL CONTINUE TO seed demo events, study plans, and attendance records for the student account (these are not part of this bugfix scope)
